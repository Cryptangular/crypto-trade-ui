# Дата: 02.06.2026

**Описание** Сегодня исправлял баги по код-ревью для компонента тостера (`StxToaster` и `ToastService`) и параллельно повторял базу по Angular перед техническим интервью

Основная задача на ревью была избавиться от `setTimeout` при удалении карточек, так как код просто «угадывал» время окончания анимации. На ревью подсказали решение завязаться на нативные события браузера. Изучил этот подход и переписал анимацию на двухэтапное удаление через `TransitionEvent`

Теперь при удалении тоста его ID сначала пушится в сигнал `animatingToastIds`, что вешает на элемент класс `.is-removing` и запускает CSS-переход. TypeScript-код больше ничего не ждет через таймеры. В шаблоне на теги `<li>` и `<aside>` повесил слушатели `(transitionend)`. Браузер сам триггерит методы `onRemoveTransitionEnd` и `onClearAllTransitionEnd` ровно в момент окончания анимации. Внутри добавил строгую фильтрацию по `event.propertyName !== 'opacity'`, чтобы код не дублировался, так как событие стреляет для каждого CSS-свойства отдельно. Только после этого данные окончательно стираются из сервиса.

После переделки анимации возникли проблемы с юнит-тестами в Vitest, которые из-за новой асинхронщины начали зависать.

**Проблемы**

1. Из-за использования `setTimeout` в первоначальной логике анимации компонента, юнит-тесты зависали на этапе ожидания `whenStable()`, так как тестовое окружение не могло дождаться завершения макротасок.

**Решение**.

1. Перенос логики анимаций на нативное событие `transitionend` по замечанию из ревью полностью устранил проблему с зависанием тестов. С помощью ИИ скорректировал тестовые сценарии в `stx-toaster.component.spec.ts`, чтобы они корректно триггерили события переходов и проверяли состояние сигналов `animatingToastIds` без использования искусственных задержек. Все тесты успешно прошли проверки.
