## Итоги Спринта 1 (Summary)

- **Что было сделано:** Начал изучать библиотеку Angular Material UI. На основе стандартного `mat-checkbox` создал изолированный кастомный компонент для повторного использования в формах приложения. Полностью интегрировал компонент в экосистему Angular Forms, реализовав интерфейс ControlValueAccessor. Теперь компонент работает как с Reactive, так и с Template-driven формами через директивы formControlName и [(ngModel)].

- **Проблемы и их решение:**
  - _Проблема:_ Изначально возникло непонимание: \_зачем и как создавать свой кастомный компонент, если можно везде напрямую использовать готовый `mat-checkbox` из библиотеки.
  - _Решение:_ В процессе разработки пришло осознание, что обертка над UI компонентом важна для дальнейшей поддержки проекта. Если в будущем мы решим сменить Angular Material на другую библиотеку (например, PrimeNG) или кардинально изменить стили/поведение чекбокса, нам придется переписывать код по всему приложению. Кастомный компонент изолирует эту логику: мы меняем код в одном месте, а для остального приложения интерфейс остается неизменным. Также это позволяет управлять валидацией и кастомной стилизацией в одном месте.

  - _Проблема:_ Возник конфликт при связывании встроенных механизмов форм Angular с логикой отключения компонента. Стандартный декоратор providers с NG_VALUE_ACCESSOR и forwardRef раздувал бойлерплейт, а изменение disabled статуса из формы конфликтовало с иммутабельными input сигналами.
  - _Решение:_ Применил продвинутый паттерн: динамически внедрил NgControl через inject() с флагом { self: true } для изоляции области видимости и избежания циклических зависимостей. Состояние disabled сделал композитным (isDisabled = (): boolean => this.disabled() || this.isFormDisabled()), объединив значение из внешнего инпута и состояние самой формы в одну геттер-функцию.
